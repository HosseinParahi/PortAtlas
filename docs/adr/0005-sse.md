# ADR 0005: SSE

- **Status:** Accepted
- **Date:** 2026-07-11
- **Scope:** Browser real-time updates

## Context

Runtime observations, scans, conflicts, reservations, collector health, and settings can change while the dashboard is open. The primary flow is server-to-browser notification; interactive commands already use REST. The UI must recover after sleep, network interruption, service restart, and dropped events without treating an event stream as the source of truth.

## Decision

Use Server-Sent Events for browser update notifications:

- GET /api/v1/events is the versioned event endpoint.
- REST resources remain authoritative snapshots.
- Events carry an event ID, schema version, event type, affected resource IDs, occurrence time, and correlation ID. They do not carry secret-bearing records.
- The UI uses events to invalidate or patch TanStack Query caches and refetches authoritative resources when event ordering or schema is uncertain.
- Support Last-Event-ID when a bounded replay buffer can satisfy it. Otherwise send a reset event that causes a snapshot refetch.
- Emit heartbeats so proxies, browsers, and the UI can detect a stale connection.
- Apply bounded per-client buffering and disconnect slow consumers rather than allowing unbounded memory growth.
- Debounce coalescible scan and observation changes while preserving reservation, lease, conflict, and security-significant events.
- Authenticate the stream as part of the browser session defined by ADR 0011 and enforce the same origin policy as the local API.

This dashboard stream is distinct from MCP Streamable HTTP in ADR 0012, although both technologies may use SSE framing.

## Alternatives considered

### WebSockets

WebSockets support bidirectional messages but duplicate command semantics already served by REST and require more connection-state and authorization machinery.

### Polling only

Polling is simple and robust but either misses the two-second update target or creates avoidable repeated full-resource traffic.

### Events as complete authoritative records

Embedding full state can reduce refetches, but increases leakage, schema coupling, stale-order risk, and replay complexity.

## Consequences

### Positive

- The protocol matches the one-way notification requirement.
- Native browser reconnection behavior and event IDs support recovery.
- REST keeps authoritative state and ordinary request validation.
- The implementation is simpler than a bidirectional socket protocol.

### Costs and risks

- Browser and intermediary connection limits need testing.
- A service restart can invalidate the replay window.
- Slow consumers and event storms need explicit bounds.
- Authentication errors must not create infinite reconnect loops.

## Verification

- Demonstrate normal runtime changes in the UI within two seconds.
- Test service restart, laptop sleep, disconnect, duplicate event, out-of-order event, stale Last-Event-ID, and replay-window overflow.
- Prove a reset event results in a correct REST snapshot.
- Load-test event bursts and slow clients with bounded memory.
- Confirm unauthorized origins and expired sessions cannot open the stream.
- Scan event payloads and logs for secrets, full environment values, and sensitive process arguments.

## Revisit triggers

- An approved browser feature requires low-latency bidirectional messaging that REST cannot provide.
- Supported deployment infrastructure cannot maintain reliable SSE connections.
- Measured event volume makes invalidation-and-refetch materially inefficient.
