# ADR 0011: Auth

- **Status:** Accepted
- **Date:** 2026-07-11
- **Scope:** Local browser, CLI, REST, SSE, and HTTP MCP authentication

## Context

Loopback binding reduces exposure but is not authentication. Other local processes, malicious browser content, DNS rebinding, copied logs, and leaked URLs can reach or reveal local endpoints. PortAtlas also has read and mutation capabilities with different risk. A full hosted identity system would be disproportionate for a single-user local MVP.

## Decision

Use local, generated credentials with channel-specific exchange:

- On first start, generate a cryptographically secure root secret with at least 256 bits of entropy.
- Store the root secret in the platform-appropriate user configuration directory with user-only permissions. Refuse to start mutating interfaces when permissions cannot be made private.
- Store verifiers or hashes for issued client credentials where protocol behavior permits; never log raw credentials.
- The browser receives a short-lived, single-use bootstrap secret from the trusted launcher. It exchanges that secret through a same-origin POST, after which the server atomically consumes it and sets a host-only HttpOnly, SameSite=Strict session cookie.
- The browser session uses an in-memory server-side record and is invalidated on service restart, explicit logout, credential rotation, or a configurable bounded inactivity timeout. No persistent browser session is issued in the MVP. Use Secure cookies whenever the deployment uses HTTPS and retain Origin and CSRF checks even on loopback HTTP.
- Bootstrap secrets never appear in server access logs, referrer-bearing query strings, browser storage, or reusable bookmarks.
- CLI and loopback HTTP MCP clients receive separate scoped bearer credentials. Initial scopes distinguish read, reserve, project administration, configuration mutation, and integration administration.
- Client tokens are displayed only at creation, stored user-only by the client, revocable independently, and auditable by credential ID rather than secret.
- Mutating requests require both an authorized scope and the application-level confirmation or dry-run policy for that operation.
- Compare secrets in constant time, rate-limit failed exchanges, rotate credentials without deleting user data, and redact every authentication header and cookie.
- Non-loopback HTTP is outside the MVP. Enabling it later requires transport security and a new identity/authorization ADR.

## Alternatives considered

### No authentication on loopback

This is convenient but leaves browser-origin, DNS-rebinding, and other-local-process attacks unchecked.

### One permanent bearer token for every client

This is simple but creates a single high-impact secret with no client attribution, independent revocation, or least-privilege scope.

### Put a permanent token in the browser URL

URLs leak through history, logs, screenshots, and referrer behavior. A one-time exchange limits exposure and moves the session to an HttpOnly cookie.

### OAuth for local single-user mode

OAuth is appropriate for remote and multi-user deployments but adds authorization-server and redirect-flow complexity not needed for the local MVP.

### Operating-system peer credentials only

Peer credentials can help on Unix sockets but do not directly authenticate ordinary browser HTTP requests and are not portable across every planned client.

## Consequences

### Positive

- A leaked client token can be revoked without rotating every integration.
- Browser JavaScript cannot read the session cookie.
- Scope and credential ID make audit events meaningful.
- The model can expand to remote authentication later without pretending loopback is already safe.

### Costs and risks

- Bootstrap delivery and token storage are packaging responsibilities.
- Local malware running as the same user may still read user-owned secrets.
- Cookie behavior on plain loopback HTTP needs browser compatibility tests.
- Credential recovery must avoid locking a user out of their local data.

## Verification

- Assert root and client credential files are user-only on supported macOS filesystems.
- Test single-use bootstrap consumption, expiry, replay rejection, logout, restart invalidation, idle timeout, and rotation.
- Verify cookies are host-only, HttpOnly, SameSite=Strict, and Secure under HTTPS.
- Reject missing, malformed, revoked, expired, incorrectly scoped, and wrong-audience bearer credentials.
- Exercise hostile Origin, DNS-rebinding-style Host, CSRF, cross-site fetch, and credential-stuffing cases.
- Scan logs, traces, diagnostics, shell history guidance, and API errors for tokens, cookies, and bootstrap secrets.

## Revisit triggers

- Any non-loopback, team, remote-agent, or multi-user deployment is approved.
- A supported browser cannot provide the required local-cookie behavior.
- Platform keychain integration becomes necessary for acceptable secret protection.
- MCP client compatibility requires a standards-based remote authorization flow.
