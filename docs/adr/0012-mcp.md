# ADR 0012: MCP

- **Status:** Accepted
- **Date:** 2026-07-11
- **Scope:** MCP server placement, transport, and compatibility baseline
- **Protocol baseline:** Model Context Protocol revision 2025-11-25

## Context

MCP is the cross-agent integration boundary for Codex, Claude-compatible hosts, and other clients. Local clients commonly support STDIO, while independently running clients benefit from Streamable HTTP. Business rules cannot diverge between transports or be copied into client-specific skills. The current MCP transport specification requires Origin validation, localhost binding, and authentication for local Streamable HTTP security.

## Decision

Implement one typed MCP protocol adapter at the application edge:

- MCP tools, resources, prompts, errors, progress, cancellation, and audit hooks call the same application services as REST and CLI.
- The adapter is part of the core service codebase, not a separate business-logic service.
- Provide a thin STDIO entry point for clients that launch an MCP subprocess. Standard output contains MCP messages only; diagnostics go to standard error.
- The STDIO entry point delegates to the running native service through a narrow authenticated local client and fails clearly when the service is unavailable. It does not open the database or reimplement collector, allocator, or policy logic.
- Provide Streamable HTTP at one loopback endpoint, authenticated with the scoped bearer credentials from ADR 0011.
- Bind Streamable HTTP to 127.0.0.1 and, only after equivalent tests, the IPv6 loopback address. Never bind to wildcard interfaces by default.
- Validate every present Origin header against an exact local allowlist. Return HTTP 403 for an invalid Origin as required by the 2025-11-25 transport specification.
- Implement POST and GET behavior, session handling, content negotiation, reconnect behavior, and cancellation against revision 2025-11-25.
- Do not implement the deprecated standalone HTTP+SSE transport for the MVP.
- Keep capability names and schemas client-neutral. Codex and Claude packages contain setup and workflow instructions only.
- Read-only tools are broadly available to read-scoped credentials. Mutating tools require explicit scopes, dry-run behavior, project boundaries, approval policy, and audit.
- AI orchestration may call only the read-only subset defined by ADR 0018 even when the human-facing MCP credential has broader scopes.

Pin the reviewed protocol revision in compatibility tests and re-review the official specification before each release.

## Alternatives considered

### STDIO only

STDIO is simple and strongly local but does not serve independently running integrations or browser-adjacent local clients well.

### Streamable HTTP only

HTTP supports multiple clients but misses the easiest installation path for many coding-agent hosts.

### Legacy HTTP plus SSE

The 2025-11-25 specification identifies Streamable HTTP as the replacement for the old HTTP+SSE transport. Supporting the legacy transport adds security and compatibility surface without an approved client need.

### A standalone MCP business service

A separate service would duplicate application rules or add another local port and lifecycle for no MVP scaling benefit.

### Client-specific skills containing logic

This would drift between clients and bypass one typed, audited permission boundary.

## Consequences

### Positive

- Clients choose the transport they support without changing semantics.
- One handler and schema set reduces policy drift.
- STDIO remains easy to configure while HTTP supports persistent local clients.
- Revision pinning makes compatibility claims auditable.

### Costs and risks

- The STDIO bridge depends on the native service lifecycle and local credential delivery.
- Two transports double connection-level test cases.
- MCP evolves and may require compatibility changes.
- HTTP sessions, Origin checks, and bearer scopes must work together correctly.

## Verification

- Run the same tool contract suite through direct handlers, STDIO, and Streamable HTTP.
- Verify STDIO emits no log or banner text on standard output and propagates cancellation.
- Test HTTP POST, GET, SSE responses, reconnect, session expiry, malformed JSON-RPC, content negotiation, and concurrent clients.
- Reject invalid Origin with 403, wildcard Host exposure, missing or wrong-scope bearer tokens, and token passthrough.
- Confirm paths remain within approved roots and mutating tools return dry-run scope before execution.
- Test supported Codex and Claude-compatible clients against the pinned 2025-11-25 baseline before release.

## Revisit triggers

- A required client supports only a different protocol revision or legacy transport.
- MCP standardizes a local IPC transport that reduces the STDIO bridge complexity.
- Remote or multi-user mode is approved.
- Protocol authorization requirements supersede the local scoped-token model.
